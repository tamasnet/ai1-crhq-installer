# Decision Record

The settled design decisions and their rationale, kept because code and docs reference the
IDs. All are resolved; there are no open questions. (The full deliberation history is in git
history of `decisions-and-open-questions.md`.)

## Core shape

| ID | Decision | Why |
|----|----------|-----|
| **D-1** | Install mechanism = **DB-direct via knex**; no REST for writes | REST can't be sandbox-intercepted; matches the satellite's established installers. |
| **D-1a / C1** | ESM `.mjs` + **hardcoded** knex import path | Only an ESM static import is interceptable by a loader hook. |
| **D-8** | Product shape = **core library + generic manifest runner** | Packages can be pure manifest *or* a 10-line `install_entry` that imports the lib; one codebase to sandbox-test. |
| **D-9** | **Jobs** (`background_jobs`) are a first-class component type | Every reference installer registers jobs. |
| **D-10** | Manifest = **`ai1-package.yaml`** — declarative `components` inventory + optional `install_entry` | Spec: `package-manifest-spec.md`. Cross-cutting implementation concerns deliberately kept OUT of it. |
| **D-12** | The utility is **both a CLI and a library** (primitives exported via `lib/index.mjs`) | One DB/fs chokepoint; `install_entry` scripts inherit sandbox-correctness for free. |
| **D-14** | The library is **opt-in**; pre-existing bespoke installers keep working unchanged | Backward compatible. |

## Configuration & paths

| ID | Decision | Why |
|----|----------|-----|
| **D-15** | Vendor-neutral env names: `INSTALL_BASE_DIR` / `INSTALL_SCHEMA`, with legacy fallback to `CRHQ_BASE_DIR` / `SANDBOX_SCHEMA` | The manifest is CRHQ-independent, so the utility's public knobs are too; fallback preserves compat with the older external harness. |
| **D-19 / C2** | `INSTALL_BASE_DIR` = the **skill-parent dir** (not the satellite root); core does `join(INSTALL_BASE_DIR, key)` | No `user-skills` knowledge in core logic — the literal survives only in the legacy shim + default. |
| **OQ-10 / C3** | `skill_path` = `db://skills/<name>` | Location-independent; doesn't bake the fs layout into the DB. |
| **D-1c / OQ-U4** | Schema configurability is **native**: `getDb()` applies `INSTALL_SCHEMA` as knex `searchPath` | No loader-hook dependency for sandboxing. |
| **OQ-U1** | Library import path = canonical absolute path (mirror of the knex.js convention) | Stable, interception-friendly, zero install magic. A `@ai1/installer` alias is a possible later DX upgrade. |

## Behavior

| ID | Decision | Why |
|----|----------|-----|
| **D-4 / C13** | Install order skills → recipes → agents → jobs → services; uninstall reverses | Dependency order. |
| **D-5 / C5** | Locked skills: auto-unlock-then-update by default; `--respect-locks` to skip | Matches canon behavior. |
| **D-22** | Skills default to **org + `locked`**; per-skill `install_type: user` or global `--install-skills-as-user` (which wins) registers unlocked `user` skills | Org skills are the norm. Only the DB registration changes — assets always land under `INSTALL_BASE_DIR` (no write access to where real org skills live). `install_type` lives in the manifest component entry, not SKILL.md. |
| **D-21** | `--type=<types>` (renamed from `--only`) is multi-valued (comma-separated/repeatable); the old `--no-agent`/`--no-job` toggles are removed | One flag expresses any type subset; `runPlan` intersects with the canonical order. `--type` reads better next to `--include`/`--exclude`. |
| **D-30** | **Strict option handling** (`lib/flags.mjs`): both CLIs reject an unsupported option or a value flag with no value (message + usage exit 2) before any side effect; `--help` prints usage (exit 0). Install's supported set = standard flags + the manifest's **declared `install_flags`** (now enforced, not just silently forwarded); a manifest may not shadow a standard flag | A typo or wrong-mode flag should fail loudly, not silently do the wrong thing. Tying the accepted package-flag set to `install_flags` keeps the existing forwarding feature while making "unsupported" well-defined. The flag contract lives in one dependency-free module so `manifest.mjs` can share it without importing the DB layer. |
| **D-20** | `--include`/`--exclude` filter by component name; regex, with metacharacter-free values as exact `^name$`; case-sensitive; zero-match = warn + exit 0; invalid regex = exit 2 | Subset installs without editing the manifest. Zero-match is "nothing to do", not a typo-failure. |
| **D-33** | `--list-installed` prints the install log (`${PACKAGES_DIR}/install.json`) as a `TYPE/NAME/VERSION/FROM` table sorted by type (canonical order) then name; `--json` emits the sorted array; empty log → notice. Standalone install-mode boolean that, like `--help`, short-circuits before manifest/validation/DB/sandbox — log-only, exit 0 (2 if unreadable). Formatting lives in `install-log.mjs` (`sortInstalled`/`formatInstalledList`) so it's unit-testable DB-free | A queryable view of installed state without a package or DB connection. Reusing the log (not the DB) keeps it dependency-free and consistent with the D-24 record; pairing the human table with `--json` serves both eyeballs and automation. |
| **D-49** | **`sync` refuses a destination that isn't inside a git repository, unless `--force`.** Both modes (plain sync and `--mirror`, including `--dry-run`) check `isInsideGitRepo(packageDir)` (in `lib/sync.mjs`: walk to the nearest existing ancestor — so a not-yet-created bootstrap dir like `<repo>/user` still counts — then shell out to `git rev-parse --is-inside-work-tree`, the same `git` dependency polaris already uses; any error / missing git / `false` → not a repo). A non-git destination throws `SyncError` (exit 1) with a `git init` / point-at-a-checkout / `--force` hint; `--force` is a new boolean flag that bypasses the check. The guard runs in the CLI **before any DB work**, so it never reaches `getDb()`. `runSync` (the library) is unguarded — the policy lives in the CLI, like the install-log write does. | sync's whole safety model is "edit the package in place; recover a bad run with git" (it's why it's git-safe / `writeIfChanged`-based). A destination with no git history has no recovery net — a wrong `--mirror` scope or a bad `--include` could clobber or delete files with nothing to revert to. Refuse-by-default-with-`--force` matches polaris's existing-checkout refusal (D-45) and the general safe-by-default posture, while `--force` keeps the throwaway/scratch case one flag away. Checking the nearest ancestor lets the common bootstrap flow (`sync.mjs --mirror <repo>/user` into a repo that doesn't yet have a `user/` dir) work without `--force`. Keeping the guard in the CLI (not `runSync`) preserves the library's purity and its many non-git temp-dir tests. |
| **D-48** | **`sync --mirror` reconciles the install log** (`${PACKAGES_DIR}/install.json`, D-24) to the live satellite for **exactly the components the mirror carries** — and only those. A mirror exports live components out of the satellite into the package, so the run returns an `installLog` delta (`installed[]` = components it added/synced/`unchanged` and faithfully wrote, with `{type,name,version?,source}`; `removed[]` = components gone from the satellite it dropped) and the CLI applies it via `updateInstallLogForMirror`: each `installed` slot is **upserted, attributed to the mirror package** (ownership transfers in place under the one-slot-per-`type:name` rule a real install follows; the original `installed_at` is preserved since the component was already live), each `removed` slot is **deleted**, and components this run didn't process — out-of-scope `--type`/`--include`/`--exclude` entries, other packages' components — are **left untouched**. `SYNC-SKIP`/`SYNC-FAIL` components (not faithfully captured) are excluded from the delta. Write-if-changed (a no-op mirror never rewrites the file) and DB-free (the delta is built from the run, the log read/written by `lib/install-log.mjs`). Dry-run reports the intended change and writes nothing. Bookkeeping only — like the install path (D-24), a log write failure warns and never fails the mirror. Plain `sync` (no `--mirror`) returns an empty delta and never touches the log. | After a mirror, `install.mjs --list-available`/`--list-installed` should agree with reality: the components the satellite's own backup package now carries ARE installed, so the log should say so. Reusing the install log's ownership-transfer model (D-24) keeps a mirror's bookkeeping identical to an install's — the mirror package is simply the package that now represents the satellite. Scoping the update to the carried set honors `--type`/`--include`/`--exclude` and never lets one package's mirror rewrite another package's claims. Returning a delta for the CLI to persist (rather than writing inside `runSync`) mirrors how `install.mjs` owns `recordInstallLog` while `runPlan` stays pure — library/test callers of `runSync` opt in. |
| **D-47** | `--list-available` is a second standalone, read-only install-mode query (the package-store-aware companion to `--list-installed`, D-33). It scans the two places install-ready packages land on a satellite — `PACKAGE_BASE_DIR` (`resolvePackageBase`, `~/packages`, where `get-package` extracts) and `REPOS_BASE_DIR` (`resolveReposBase`, `~/repos`, where `polaris init` clones Client Repos whose `platform/`+`user/` are packages) — enumerates every component each discovered package declares, then joins `${PACKAGES_DIR}/install.json` to label each `type:name` **per version** (`type:name@version` — a component present at multiple versions across packages lists once per version): **available** (in a package at this version, not the installed version), **installed** (the version the log pins, still in a package), **missing** (the version the log pins, in NO package — source gone). Output is a `STATUS/TYPE/NAME/VERSION/PACKAGE/LOCATION` table (or the `rows` array under `--json`); same-version copies from different packages merge into one row whose `PACKAGE`/`LOCATION` list **every** provider so duplicate sources are visible (`missing` rows fall back to the log's package + an em-dash location). Like `--list-installed` it short-circuits before manifest/validation/DB/sandbox. Discovery walks ≤3 levels and **stops at each package root** (component subtrees aren't packages), skipping hidden dirs/`node_modules`/symlinks; a per-package manifest-load failure or an unreadable log is a **warning, not fatal** (exit `2` reserved for an unexpected scan fault). Logic + the pure formatters live in `lib/list-available.mjs`, unit-testable DB-free against the test fixtures as a store | The install log answers "what did I install"; this answers the operator's next question — "what *can* I install, and is what I installed still backed by a local package?" — without a DB connection. Scanning the get-package and polaris stores (not the DB) keeps it dependency-free and reuses the existing `*_BASE_DIR` resolvers, so the two discovery surfaces the satellite already writes to become directly queryable. Surfacing **missing** turns a silent drift (installed component whose package was removed) into a visible state, and listing all provider locations makes accidental duplicate packages diagnosable. Resilient-by-default (warn, don't abort) because a single malformed package shouldn't blind the operator to every other one. |
| **D-34** | **Component versions are positive integers** (not semver) and round-trip through CRHQ's `*_versions` tables (`skill_versions`/`recipe_versions`/`agent_versions`; `version_num` int, `UNIQUE(<fk>,version_num)`, current = `MAX`). Required for skills/services, optional for recipes/agents, n/a for jobs. Install upserts `version_num` = the declared integer (`changed_by='ai1-installer'`, merge → idempotent); backup reads `MAX(version_num)` as the pin (no history → `1` + warn). A non-incrementing version **warns and continues**. Uninstall drops history via the FK cascade (mirrored explicitly in the FK-less sandbox, which also clones the two new version tables). Logic lives in `version-history.mjs`. The **package** `version` stays a free-form label; `hub_version`/`org_version`/`source_version` are left untouched | Integers map 1:1 to CRHQ's `version_num`, so the package's number and the satellite's number stay in lockstep across install→backup→reinstall — which semver strings couldn't. The `*_versions` tables (not the hub/org sync columns) are the canonical per-component version store. Warn-not-block keeps installs frictionless while still surfacing accidental downgrades. |
| **D-35** | The manifest's optional `installer` field is a **plain positive integer** (not a semver range) = the minimum installer version the package needs, with an **implicit `>=`**. The installer's own integer version is `INSTALLER_VERSION` (in `manifest.mjs`; currently `1`, tracking the `package.json` major); a package whose `installer` exceeds it is rejected at load with a `ManifestError` | Matches the integer-versioning model (D-34) — no semver anywhere in the manifest. A bare integer with implicit `>=` is the simplest expression of "needs at least this generation"; failing fast at load beats a confusing mid-install error when the installer is too old. |
| **D-23** | Agent manifest fields follow the common pattern: `name` (the canonical identifier, stored as CRHQ `agents.key`) + `display_name` (stored as `agents.name`); the former `key` field is rejected with a pointer to the rename | Every component type shares the `name`/`description` shape, so filters, summaries, and docs treat all types uniformly. |
| **D-50** | **An agent is a DIRECTORY component (`agents/<key>/`) with an `AGENTS.md`, exactly like a skill's `<key>/SKILL.md` — its "brain".** `AGENTS.md` frontmatter → the agent record + body → `instructions` (the D-32 fields are unchanged); the **whole directory** (AGENTS.md included) copies to `AGENT_BRAINS_DIR/<key>` on install — the agent-side analog of a skill's `INSTALL_BASE_DIR/<key>` (new `resolveBrains()` in `context.mjs`, `ctx.BRAINS`, sandbox redirect; same env→`CRHQ_BASE_DIR`-relative→`documents/agent-brains` resolution as `INSTALL_BASE_DIR`). **(a) Clean break:** a flat `agents/<name>.md` is rejected at manifest load with a pointer to the directory form (matches the D-32 precedent — pre-release, one example agent). The manifest `components.agents[].path` is now a directory; `deriveName`/`sourceOf`/`DEFAULT_PATH` treat agents like skills (`AGENTS.md` is the install-log `source`). **(b) Uninstall PRESERVES the brain folder** — only the row + joins + version history are removed — because the brain accumulates agent-authored runtime state (`activity/`, `data/`, `_backup/`) an uninstall must not destroy (a deliberate, documented exception to the §7 clean-lifecycle guarantee). `copyTree` is byte-idempotent and never deletes, so a reinstall over a live brain is drift-free and keeps that runtime state. **(c) Backup/`--mirror` excludes runtime dirs:** the capture regenerates `AGENTS.md` from the row (skip-on-copy, the D-41 SKILL.md fix) and copies the brain tree EXCLUDING `AGENTS.md` + the runtime/transient top-level dirs in `AGENT_BRAIN_EXCLUDE` (default `activity,_backup,.scratch,memory`; set empty to capture everything), so a distributable package carries the seed brain, not one agent's logs. **(d) Shared `_shared/` brain files are out of scope** for this iteration. | Real satellites store an agent's behavioural detail as a folder of files (`documents/agent-brains/<key>/`) the DB loader points at, not inline in `instructions`; the package format must round-trip that. Modeling the agent exactly like a skill (directory + Markdown index + configurable install root) reuses the whole skill machinery (`copyTree`, version round-trip, install log `source`) and keeps one mental model. Preserving the brain on uninstall is the one place an agent must diverge from a skill — a skill dir is shipped code, a brain dir mixes shipped seed with live runtime data — so destroying it would lose user work. Excluding runtime dirs from capture keeps backups clean and distributable without a per-agent allowlist. Clean break over a dual-format parser is the same call D-32 made and the project is still pre-release. |
| **D-24** | **Install log** at `${PACKAGES_DIR}/install.json` (default `~/packages`): a **flat list**, one entry per component (`type:name`) `{type, name, version?, package, package_version, source, installed_at}` (source = the component's manifest file relative to its package root). One slot per component mirrors the DB's one-row-per-name rule, so re-installing a component (newer version of the same package, or a different package) **transfers ownership by overwriting that slot** — no duplicates, and a partial upgrade reads as mixed `package_version`s. Never written in dry-run/status; uninstall deletes the entry; `--sandbox` redirects `PACKAGES_DIR` to a throwaway dir; a write failure warns but never fails the install | A queryable record of what's installed, independent of the DB. Component-keyed (not package-keyed) because the DB allows exactly one component per name — making that the log's primary key keeps an impossible "two packages own one skill" state unrepresentable and reduces ownership transfer to a plain upsert. Removal-not-tombstoning keeps it a statement of current state, not a history (git/DB hold history). |
| **OQ-U2** | `install_entry` runs on **all modes** (install/uninstall/status), receiving the mode | Keeps teardown symmetric; the hook decides what to do. |
| **OQ-U3** | `--json` machine-readable report | Cheap; enables automation over the verdict taxonomy. |
| **D-7 / C7** | `--dry-run` = zero side effects; output contains "would…" and >200 bytes | It's the built-in pre-flight check. |

## Backup / sync (`sync.mjs --mirror` — the reverse of install)

> **D-41 consolidated `backup.mjs` into `sync.mjs --mirror`** (see the D-41 row below). The scope,
> skip, version-table, and dry-run rationale in D-25/D-28/D-29/D-31 still hold and now describe
> `--mirror`; the CLI/identity/atomicity specifics of **D-26/D-27 are superseded by D-41** (package-dir
> positional, in-place git-recoverable writes, integer package version). References to `backup.mjs` /
> `BACKUP_BASE_DIR` / `BACKUP-*` verdicts below are historical — read them as `sync --mirror` / the
> `<package-dir>` argument / the `SYNC-*` verdicts.

| ID | Decision | Why |
|----|----------|-----|
| **D-25** | Backup scope = active `org`+`user` skills, active recipes, non-system active agents, non-system jobs; platform `system` components and inactive rows are out of scope; **services excluded in v1** | System components are restored by the platform, not a package; the manifest can't express `is_active:false` (restoring one would silently re-activate it); services aren't DB-resident and their source of truth is the original package. |
| **D-26** | Output = `${BACKUP_BASE_DIR}/<name>/` (env, default `~/backups`; positional arg overrides), **overwritten in place** each run — built in a staging dir and swapped only after the generated manifest passes the real `loadManifest()` | One current snapshot, user-managed rotation (git/rsync hold history). The stage-validate-swap makes a failed backup unable to clobber the previous good one, and guarantees the output is installable at parse level. |
| **D-27** | Package identity: name = `<satellite-id>-backup` (`SATELLITE_ID` env, else hostname minus `crhq-`; `--name` overrides); **package** version = date-based label (`YYYY.M.D`), minted at the CLI entry. Component versions come from the version tables (D-34): a skill with no `skill_versions` history pins `1` + warning | Multi-satellite backups distinguishable by default; the date labels the snapshot. The package version is a free-form label, distinct from the integer component versions. `1` keeps the manifest valid (skill pins are required) without inventing a version. |
| **D-28** | Lossiness is explicit, never fatal: a component the format can't express (non-script job, script outside `INSTALL_BASE_DIR`) → `BACKUP-SKIP` (severity 0) + warning | Backup is best-effort over a live DB that contains more shapes than the manifest format models; silent omission would read as "covered", aborting would block the rest of the backup. (The agent fields that D-28 originally listed as lossy — `instructions`, `capabilities`, `system_prompt_path`, `provider` — now round-trip via the `.md` format, D-32.) |
| **D-29** | YAML **emission** is hand-rolled (`dumpYaml`, ~60 lines): plain scalars only when provably safe, JSON-double-quoted otherwise (valid YAML); round-trip tested against the vendored parser | Extends D-6 — the vendored bundle only exports `parse`, and regenerating it for `stringify` adds weight for shapes we fully control. JSON-escape fallback makes it correct by construction. |
| **D-31** | Backup supports `--dry-run`: the full discovery/scope/export pipeline runs — including the D-28 skip rules, warnings, and per-component verdicts — with **zero filesystem writes** (the `export*` primitives thread `ctx.DRY_RUN` into the fs helpers); the generated manifest is validated in memory and the previous backup is untouched. Still no `--status`/`--uninstall`/`--sandbox` | Lets the operator see what a backup would contain and test `--type`/`--include`/`--exclude` combinations without overwriting the current snapshot. Reuses install's existing dry-run plumbing (one convention, C7 `would` markers) instead of inventing a backup-specific preview mode. |
| **D-32** | Agents are a **content-bearing** component: YAML frontmatter (config) + Markdown body (`instructions`). _(The single-file `agents/<name>.md` carrier is **superseded by D-50** — the agent is now a directory `agents/<key>/` with an `AGENTS.md`; the frontmatter→fields and body→`instructions` mapping below is unchanged.)_ Frontmatter now also carries `provider`, `system_prompt_path`, and `capabilities` (jsonb). Install sets each only when present (else DB default; `capabilities` is `JSON.stringify`d) and drift-checks them; backup emits each only when non-default, so the round trip is lossless and idempotent. **Clean break — the former `.yaml` agent format is no longer parsed.** | These fields are needed (exposed while building backup, which previously dropped them under D-28). An agent's `instructions` is Markdown, so the agent belongs with skills/recipes as a frontmatter+body component rather than flat YAML. The project is pre-release with one example agent, so a clean switch beats carrying a dual-format parser. |
| **D-44** | **An exported-but-unchanged component reports `SYNC-UNCHANGED` ("unchanged"), not `SYNC-OK` ("synced").** Phase 2 now tracks per-component `entryChanged` (a byte written — the `changed` flag the `export*` fns return — OR a manifest-only move: version pin bump / skill `install_type` reconcile) and only emits `synced` (+ a per-line log) when it's true; otherwise `unchanged`, logged at no level. `counts` gains an `unchanged` tally. Applies to plain sync and `--mirror` alike. | The export layer is byte-idempotent (`writeIfChanged`), so a clean re-run touches nothing — but the verdict still claimed `synced` for every component, making a no-op run look like work and burying real diffs. Reusing the existing `changed` signal (added for the D-41 version-bump) makes the report match what actually happened and keeps repeated mirrors quiet. |
| **D-43** | **A `--mirror` bootstrap names the package via a shared `satellitePackageName()` helper** (`lib/identity.mjs`, also exporting `resolveSatelliteId()`). The heuristic: start from the satellite id (`SATELLITE_ID` env, else hostname minus `crhq-`, per D-27/D-37) → drop a leading `myzone-` → ensure a leading `ai1-` (e.g. `myzone-tamas` → `ai1-tamas`, `ai1-foo` → `ai1-foo`). `identity.mjs` is **dependency-free** (only `os` + env) so DB-free consumers (the remote client, an upcoming publish utility) can import it without the knex layer. A plain-sync `--add` bootstrap still uses the directory basename. | The satellite's own backup/mirror package should carry a stable, satellite-derived name rather than whatever the working directory happens to be called, and the same name will be needed by a forthcoming utility (e.g. publishing the package to the hub) — so the derivation is factored into one shared, lightweight, separately-tested helper instead of being duplicated. Keeping it out of `context.mjs` avoids dragging the DB/log layer into network-only callers. |
| **D-42** | **Mirror auto-adds only `user` skills** (`discover()` filters `skill_type:'user'`, not `org`/`store`/`system`). This narrows *only the auto-add candidate set* — removal still uses `findRow` (exists by name + active, any type) and sync-existing still handles any type, so an `org`/`store` skill already listed in a manifest is synced and kept, never auto-purged. Recipes/agents/jobs auto-add scope is unchanged. | `org`/`store`/`system` skills are owned by their own source packages (and the platform), not authored on the satellite — a satellite backup should capture the user's own work, and those skills are restored from where they came from. Keeping removal/sync type-agnostic means a manifest an author deliberately curated with an org skill isn't fought by the next mirror run. |
| **D-41** | **`backup.mjs` is removed and folded into `sync.mjs` as a `--mirror` mode** (`backup.mjs` + `lib/backup.mjs` deleted; logic now in `lib/sync.mjs`, reusable as `runSync({mode:'mirror'})`). The two were already ~90% the same (shared `export*`, manifest read/write, version logic); after this they differ only by behavioral flags. **(a) CLI**: `--mirror` takes the destination as the `<package-dir>` positional like plain sync — `BACKUP_BASE_DIR`, `--name`, and the `<satellite-id>-backup` default are dropped; an empty/new dir bootstraps the manifest from the dir name (`version: 1`). **(b) Reconcile, don't regenerate**: within the `--type`/`--include`/`--exclude` scope, mirror **adds** curated-live components missing from the manifest, **syncs** the ones present, and **removes** entries + their files whose component is gone from the satellite (liveness = `findRow`; auto-add set = `discover`). **(c) In place, git-recoverable**: writes via `writeIfChanged` + targeted deletes (no staging/swap — D-26 superseded); recover a bad run with git. **(d) Fidelity**: a new skill preserves its live `install_type` (a user skill stays unlocked) so a restore reproduces the satellite; `--normalize` ships the locked `org` default (the plain-sync `--add` behavior). **(e) Version**: the package `version` is an **integer**, +1 only on a content-changing run (a no-op leaves it; D-27's date label superseded), driven by a new `changed` flag the `export*` fns return. Unrepresentable live components (D-28) are `SYNC-SKIP`, never auto-added. | "Consolidate as much as possible" once sync existed: one mental model (point at a package dir), one library, one test surface. Reconcile-over-regenerate makes mirror diff-minimal and lets `runSync` own version bumping for existing components (backup no longer does). Conservative removal (only when truly absent) + curated add (only the standard org/user/active/non-system set) means a filtered or partial backup never silently purges out-of-scope package content. In-place writes match sync's git-working-copy model and drop the staging machinery; fidelity-by-default makes a mirror a faithful restore while `--normalize` keeps the distributable path. The integer-bump-on-change keeps a meaningful monotonic snapshot number without churning the manifest on clean runs. **Bugfix surfaced here:** `exportSkill`'s `copyTree` was copying the installed `SKILL.md` (which it then regenerates from the DB), so the file flip-flopped and every run looked "changed" — `copyTree` gained a `skip` predicate and `exportSkill` now skips it. |

## Remote (`remote.mjs` — Ai1 Platform Hub client)

| ID | Decision | Why |
|----|----------|-----|
| **D-36** | A third CLI, `remote.mjs`, is the satellite's side of the **Ai1 Platform Hub** (`ai1-platform-hub/`): a **subcommand** dispatcher (`remote.mjs <subcommand>`) rather than a flag-driven single-action tool like install/backup. First subcommand `register`; config/state poll, management instructions, and package download follow. Logic in `lib/remote.mjs`, entry in `remote.mjs` — the install/backup split. **DB-free and network-only**: the hub is reached over HTTPS with Node's built-in `fetch` (no runtime deps, per D-6) | The hub is a fundamentally different axis from install/backup (talk to a remote service vs. mutate the local DB), and it has several distinct verbs — subcommands model that better than one mode plus flags. Keeping it DB-free means it needs no sandbox and can run anywhere the satellite does. |
| **D-37** | `register` → `POST {hub}/remote/register` (the hub's bootstrap-token-authenticated self-enroll). Inputs resolve **flag → env**: hub `--hub=`/`AI1_HUB_URL`(/`HUB_URL`); bootstrap `--token=`/`AI1_BOOTSTRAP_TOKEN`(/`BOOTSTRAP_TOKEN`); `remote_id` `--remote-id=`/`SATELLITE_ID`/hostname-minus-`crhq-` (the D-27 convention). `remote_type` defaults to `crhq-satellite`, `schema_version` to `1` (both overridable). A missing hub/bootstrap is a **usage error (exit 2)**; option validation mirrors D-30 (strict, per-subcommand) | Flag-then-env keeps secrets out of argv when wanted while staying scriptable; reusing the `SATELLITE_ID` derivation makes a satellite's remote identity match its backup identity by default. The hub already defaults `remote_type`/`schema_version`, so these are just explicit, overridable client-side defaults. |
| **D-38** | The minted per-remote **token** + identity are persisted to `${REMOTE_BASE_DIR}/id.json` (env, default `~/remote`), written **atomically** (temp+rename) at mode **0600** — it holds a credential. Stored fields: `remote_id`, `token`, `remote_type`, `schema_version`, `hub_url`, `registered_at`. The lifecycle **`status` is deliberately NOT stored** — it is the hub's to own (changes server-side on approve/reset/revoke) and would only go stale locally; it is surfaced to the operator at registration time instead | The token is returned by the hub exactly once, so id.json is its only copy — 0600 + atomic write protect it. Persisting hub-owned mutable state (status) locally invites drift; the durable facts are the identity and the credential. `REMOTE_BASE_DIR` mirrors the `INSTALL_BASE_DIR`/`BACKUP_BASE_DIR` env convention. |
| **D-39** | Registration **refuses to overwrite an existing `id.json` unless `--force`** (exit 1 with the stored `remote_id`). Hub rejections map to clear messages: `401` → bad bootstrap token; `409` → cannot register (with a "an operator must reset it" hint); unreachable host → "could not reach hub". | The hub itself 409s a re-register of a provisioned remote, and overwriting would discard the only copy of the existing token — so the safe default is refuse-and-tell, with `--force` as the explicit escape hatch. Translating the hub's error envelope into operator-actionable messages keeps the failure modes self-explanatory. |
| **D-40** | `get-package` (`--name= --version= [--keep-download] [--json]`) is the **package-download** verb foreseen in D-36. It resolves a download URL via `GET {hub}/remote/package?name=&version=&format=json` (Bearer token from id.json) — the hub returns `{ url, expires_at }`, a short-lived **pre-signed GCS** GET URL — then **fetches that URL with no Authorization header** (it self-authenticates via its query string and is cross-host; sending the per-remote token there would leak it), streams the archive to `${DOWNLOAD_BASE_DIR:-<system temp>}/<name>@<version>.<ext>` (0600), **shells out to `tar`/`unzip`** to extract it into `${PACKAGE_BASE_DIR:-~/packages}/<name>@<version>` via **stage→swap**, and deletes the archive unless `--keep-download`. `--version` must be a positive integer; the hub `404` ("(name, version) not registered") → exit 1; same 401/403 mapping as the other authed subcommands; the signed URL is a credential and is **never echoed** to stdout/`--json`. The extracted dir is a ready `install.mjs <dir>` input. | Asking the hub for `format=json` (rather than following its default `302`) lets the client own the download — pick the archive path, stream it, and report a structured result — while keeping the per-remote token off the GCS hop. Two env dirs separate the disposable transfer artifact (`DOWNLOAD_BASE_DIR`, default system temp) from the durable install input (`PACKAGE_BASE_DIR`, the `~/packages` root install.json already lives in), matching the `*_BASE_DIR` convention (D-38). Stage→swap reuses backup.mjs's overwrite-in-place pattern so a failed extract never leaves a half-written package where install.mjs would pick it up; shelling out to `tar`/`unzip` keeps the zero-runtime-deps rule (D-6) intact, the same shell-out service.mjs already uses for builds. |

## Polaris (`polaris.mjs` — GitHub Client-Repository client)

The parent–client repository model this implements is described in `docs/repo-methodology.md`.
A satellite's **Client Repository** pairs a `platform/` directory (an Ai1 Package fed from the shared
platform *parent* repo via git subtree) with a `user/` directory (an Ai1 Package of this satellite's
own customer/user content). polaris bridges that repo and the live satellite: `install.mjs <repo>/platform`
+ `install.mjs <repo>/user` load content **in**; `sync.mjs --mirror <repo>/user` pushes live user
edits back **out** to the `user/` package only.

| ID | Decision | Why |
|----|----------|-----|
| **D-45** | A fourth CLI, `polaris.mjs`, is the satellite's client for its **GitHub Client Repository** — a **subcommand** dispatcher (`polaris.mjs <subcommand>`) like `remote.mjs`, not a flag-mode tool. Logic in `lib/polaris.mjs`, entry in `scripts/polaris.mjs`. **DB-free**: it shells out to `git` and reuses the hub client's `fetchGithubToken()` (no new network code, no runtime deps). First subcommand **`init`**: clone the Client Repository into `${REPOS_BASE_DIR}/<repo>` (env, default `~/repos` — the `*_BASE_DIR` convention). The repo is `<owner>/<repo>` on github.com: **owner** resolves `--owner=` → `AI1_GITHUB_OWNER` → the **`MyZone-AI`** default constant; **repo** resolves `--repo=` → `satellitePackageName()` (D-43). owner/repo are charset-guarded (`[A-Za-z0-9._-]`, no bare `.`/`..`) so they can't escape `REPOS_BASE_DIR` or steer the URL off github.com. init **always errors if the destination exists** (no `--force`) — a re-clone would discard unsynced `user/` edits — and the check runs **before** the token call (fail fast, no network). The remote's **default branch** is cloned (no branch selection in v1). Strict per-subcommand option validation mirrors D-30 (reuses `UsageError`); exit codes: `0` ok · `1` `PolarisError`/`RemoteError` · `2` usage. | The GitHub repo is a different axis from both the local DB (install/sync) and the hub (remote) — and it has several foreseeable verbs (clone, pull/update, push user changes, manage the subtree), so a subcommand CLI is the right shape, consistent with `remote.mjs`. Reusing `fetchGithubToken()` means the clone authenticates with the **same** per-remote token the hub already mints — one credential path, so the satellite must simply be registered first. A hardcoded `MyZone-AI` default with flag/env override fits the ~40-client single-org reality (the methodology doc) while staying redirectable. Always-error (no `--force`) is the safe default for a checkout that may hold un-pushed customer work; manual removal is the deliberate escape hatch. |
| **D-46** | The clone **credential is injected via git's environment-based config** (`GIT_CONFIG_COUNT`/`GIT_CONFIG_KEY_0`/`GIT_CONFIG_VALUE_0`, git ≥ 2.31) as a host-scoped `http.https://github.com/.extraheader` HTTP Basic header (`x-access-token:<token>`), **not** via the clone URL and **not** via argv. `GIT_TERMINAL_PROMPT=0` makes a bad/missing token fail fast instead of hanging on a prompt. Consequence: the token never appears in `ps`/argv, and because command-scoped env config is **not** written to the cloned repo's `.git/config`, `origin` is left as the clean, tokenless `https://github.com/<owner>/<repo>.git`. The token (and the derived header) are **never logged**. | The token is a live credential. Putting it in the clone URL would persist it into `.git/config` (a durable on-disk leak); putting it in `-c` argv would expose it in `ps`. The env-config approach — the same mechanism GitHub Actions' checkout uses — avoids both: nothing is persisted and nothing is in the process table, leaving only the owner-readable `/proc/<pid>/environ` exposure acceptable on a single-tenant satellite. A clean tokenless `origin` is also the correct end state for later push/pull verbs (which re-supply auth). |
